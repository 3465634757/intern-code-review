# 测试实习生 - 岗位专项评审标准

> 本文件为岗位叠加层，继承 `_base-template.md` 的通用评审框架。
> 以下内容占据总分的 **15%（岗位专项维度）**。

## 岗位专项评审维度

### 1. 测试用例质量 (3分)

- 用例描述是否清晰（Given-When-Then 或等效结构）
- 是否覆盖正常路径、边界值、异常路径（至少各1条）
- 测试数据构造是否合理（不依赖外部真实数据）
- 是否有数据驱动测试意识（参数化测试）
- 用例粒度是否合适（一个测试方法验证一个行为）
- 命名是否能一眼看出在测什么（`test_should_return_error_when_input_is_empty`）

### 2. 自动化测试能力 (3分)

- 测试框架使用是否得当（JUnit/pytest/Jest/Cypress/Appium）
- 断言是否充分（避免空断言、只断言非null）
- 测试是否独立（无顺序依赖、可单独运行、不互相污染）
- setUp/tearDown 是否正确使用（资源清理、环境隔离）
- Mock/Stub 使用是否合理（不过度Mock导致测试无意义）
- CI 集成意识（测试是否能在 CI 环境跑通）

### 3. 缺陷发现能力 (3分)

- 本周发现的有效 Bug 数量和严重程度
  - P0/P1 级 Bug：每个 +5
  - P2 级 Bug：每个 +3
  - P3 级 Bug：每个 +1
  - 无效 Bug（非问题或无法复现）：每个 -2
- Bug 报告质量：
  - 标题是否清晰概括问题
  - 复现步骤是否完整（1-2-3步骤）
  - 是否附带截图/日志/视频
  - 是否标注了预期结果 vs 实际结果
  - 是否指出了影响范围

### 4. 测试策略与工具 (3分)

- 是否制定了合理的测试计划（针对需求/功能点）
- 是否关注了非功能测试（性能、安全、兼容性）
- 是否善用工具提升效率（Fiddler/Charles/Postman/浏览器DevTools）
- 测试脚本是否可复用（抽取公共方法、Page Object 模式）
- 是否有测试数据管理方案（工厂模式、fixtures）
- 测试报告是否清晰（通过率、失败原因、覆盖率）

### 5. 质量意识 (2分)

- 是否在需求阶段就参与评审并提出测试角度的建议
- 是否关注用户体验（从真实用户角度测试）
- 是否主动与开发沟通确认是Bug还是Feature
- 是否关注回归测试（修复后验证相关功能）
- 是否记录了测试用例到用例管理系统

## 测试专项扣分示例（供AI参考）

| 问题 | 扣分 | 理由 |
|------|------|------|
| Bug报告无复现步骤 | -5 | 无法定位问题 |
| 测试用例无断言 | -8 | 形同虚设 |
| 测试之间有顺序依赖 | -5 | 不可靠 |
| 未发现明显P0级Bug | -8 | 测试不充分 |
| 提交大量无效Bug | -3 | 浪费开发时间 |
| 测试脚本不可复用 | -3 | 效率低下 |
| 只测正常路径不测异常 | -5 | 覆盖不全 |
