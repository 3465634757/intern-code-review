# 数据/算法实习生 - 岗位专项评审标准

> 本文件为岗位叠加层，继承 `_base-template.md` 的通用评审框架。
> 以下内容占据总分的 **25%（岗位专项维度）**。

---

## 岗位专项评审维度

### 1. 数据处理与ETL质量（20分）

- **数据源接入可靠性**：
  - 是否处理了异常数据和脏数据
  - 空值填充策略是否合理
  - 数据格式转换是否正确
  - 是否有数据校验和监控
- **数据清洗完整性**：
  - 去重逻辑是否正确
  - 异常值处理是否合理
  - 数据标准化/归一化是否规范
  - 缺失值处理策略是否合适
- **数据质量校验**：
  - 是否有 schema 校验
  - 是否有行数/数据量校验
  - 字段完整性和一致性校验
  - 是否有数据质量监控告警
- **大数据处理性能**：
  - 是否考虑分批处理，避免全量加载到内存
  - 是否使用流式处理处理大数据量
  - 是否有性能优化（分区、索引、并行计算）
- **幂等性保证**：数据管道重跑是否会产生脏数据，是否有幂等设计
- **数据血缘**：是否有数据血缘记录，数据从哪来到哪去是否清晰

---

### 2. 模型与算法设计（20分）

- **特征工程质量**：
  - 特征选择是否合理，是否有特征重要性分析
  - 缺失值处理方式是否得当
  - 编码方式选择是否合理（OneHot/Label/Target Encoding等）
  - 特征缩放/归一化是否正确
  - 是否有特征交叉和组合特征
- **模型选择依据**：
  - 为什么选择这个算法，是否有对比实验
  - 是否考虑了业务场景和数据特点
  - 模型复杂度和可解释性的权衡是否合理
- **数据集划分正确性**：
  - 训练/验证/测试集划分是否合理
  - 是否有数据泄漏问题（特征工程时用了测试集信息）
  - 时间序列数据是否按时间划分，避免未来函数
- **超参数调优**：
  - 是否有系统的超参数搜索记录
  - 是否使用了网格搜索/随机搜索/Bayesian优化
  - 调优过程是否可复现
- **评估指标全面性**：
  - 不只看准确率，还看精确率、召回率、F1、AUC等
  - 是否考虑了类别不平衡问题
  - 是否有业务指标评估
  - 是否有混淆矩阵、ROC曲线等分析
- **过拟合/欠拟合识别与处理**：
  - 是否能识别过拟合和欠拟合
  - 是否有相应的处理策略（正则化、增加数据、模型调优等）

---

### 3. 代码工程化质量（20分）

- **模块化设计**：
  - 数据处理管道是否模块化，而非一个巨大的脚本
  - 函数职责是否单一，可复用性如何
  - 是否有清晰的模块划分和目录结构
- **配置化管理**：
  - 硬编码的路径、阈值、参数是否提取为配置文件
  - 配置是否区分环境（开发/测试/生产）
  - 是否有配置校验
- **数据与模型版本管理**：
  - 是否有数据集版本管理（DVC等）
  - 模型版本是否有记录和管理
  - 实验数据是否可追溯
- **实验记录与追踪**：
  - 是否使用 MLflow/W&B 等实验追踪工具
  - 实验参数、指标、模型是否都有记录
  - 实验记录是否完整可查
- **可复现性保证**：
  - 随机种子是否固定
  - 环境是否锁定（requirements.txt/conda.yml/Dockerfile）
  - 依赖版本是否明确
  - 其他人是否能复现实验结果
- **代码注释与文档**：
  - 算法选择理由是否有说明
  - 参数含义是否有注释
  - 数据假设和前提是否有说明
  - 关键逻辑是否有注释

---

### 4. SQL与数据查询能力（20分）

- **查询效率优化**：
  - 是否避免了全表扫描
  - 是否合理使用分区
  - 是否有索引优化意识
  - 大查询是否有优化手段
- **窗口函数与CTE使用**：
  - 窗口函数使用是否正确、熟练
  - CTE（公用表表达式）使用是否合理
  - 是否能写出复杂的分析查询
- **预聚合与物化视图**：
  - 大查询是否考虑预聚合
  - 是否使用物化视图提升查询性能
  - 是否有增量计算的意识
- **数据口径一致性**：
  - 指标定义是否清晰、有文档
  - 是否与业务方对齐了数据口径
  - 不同报表的同一指标是否一致
- **数据权限与安全**：
  - 敏感字段是否脱敏处理
  - 是否有数据访问权限控制
  - 是否符合数据安全规范

---

### 5. 数据可视化与输出（20分）

- **图表选择合理性**：
  - 是否根据数据类型和分析目的选择合适的图表
  - 图表类型是否准确传达信息（折线图看趋势、柱状图看对比、饼图看占比等）
- **标注完整性**：
  - 标题、坐标轴标签、单位、图例是否完整
  - 关键数据点是否有标注
  - 数据来源是否说明
- **结论支撑性**：
  - 结论是否有数据支撑，不说"感觉"，说"数据显示"
  - 分析是否深入，不只停留在表面
  - 是否有洞察和建议
- **异常波动归因**：
  - 数据异常波动是否有归因分析
  - 是否能定位问题原因
  - 是否有改进建议
- **输出格式友好性**：
  - 输出格式是否便于下游使用（CSV/Parquet/API等）
  - 数据格式是否规范
  - 是否有数据字典说明

## 数据专项扣分示例（供AI参考）

| 问题 | 扣分 | 理由 |
|------|------|------|
| 训练集数据泄漏到测试集 | -10 | 实验结果完全不可信，严重问题 |
| 全量数据加载到内存导致OOM | -8 | 大数据处理的基本常识错误 |
| 硬编码绝对路径，无法移植 | -4 | 工程化差，代码无法在其他环境运行 |
| 只看准确率，不评估其他指标 | -5 | 评估不全面，可能误导结论 |
| 没有实验记录，无法追溯 | -4 | 实验不可复现，团队协作困难 |
| SQL 全表扫描，没有分区裁剪 | -6 | 查询性能差，浪费资源 |
| 缺失值直接删行，不做分析 | -3 | 数据损失，可能引入偏差 |
| 没有随机种子，结果不可复现 | -4 | 实验结果不稳定，无法对比 |
| 图表没有标题和坐标轴标签 | -2 | 可视化不规范，信息传达不清 |
| 数据口径不一致，结果有歧义 | -5 | 数据可信度低，可能误导决策 |
