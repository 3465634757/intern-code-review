 # 运维/DevOps实习生 - 岗位专项评审标准

> 本文件为岗位叠加层，继承 `_base-template.md` 的通用评审框架。
> 以下内容占据总分的 **25%（岗位专项维度）**。

---

## 岗位专项评审维度

### 1. CI/CD 流水线设计与优化（20分）

- **配置可读性**：流水线配置文件是否清晰易读，YAML 格式是否规范，是否有注释说明
- **阶段划分合理性**：构建阶段划分是否合理（lint → test → build → deploy 等）
- **缓存策略**：是否有依赖缓存、构建缓存策略，能有效减少构建时间
- **环境隔离**：部署是否有环境隔离（dev/staging/prod），不同环境配置是否分离
- **回滚方案**：是否有完善的回滚方案（蓝绿部署、金丝雀发布、版本回退）
- **敏感信息管理**：密钥、密码等敏感信息是否使用 Secrets 管理，不硬编码在配置中
- **流水线效率**：是否考虑了并行构建、分布式构建等提升效率的手段
- **失败处理**：流水线失败时是否有清晰的错误提示和告警机制

---

### 2. 容器化与编排（20分）

- **Dockerfile 最佳实践**：
  - 是否使用多阶段构建减小镜像体积
  - 是否合理使用 .dockerignore 排除无关文件
  - 是否不以 root 用户运行容器
  - 基础镜像版本是否锁定，避免 latest 标签
  - 是否合理分层，利用缓存加速构建
- **Docker Compose / K8s 配置规范性**：配置文件是否规范，是否有注释说明
- **资源限制配置**：是否设置了合理的 CPU/内存资源限制和请求
- **健康检查配置**：是否配置了 liveness/readiness probe，健康检查是否合理
- **日志收集**：容器日志是否打通了收集链路，日志格式是否结构化
- **存储管理**：持久化存储配置是否合理，数据备份是否考虑
- **网络配置**：服务发现、负载均衡、网络策略配置是否合理

---

### 3. 监控告警与可观测性（20分）

- **基础指标监控**：是否配置了关键的系统指标监控（CPU、内存、磁盘、网络）
- **应用层监控**：是否有应用层面的监控（错误率、响应时间、QPS、业务指标）
- **告警规则合理性**：
  - 阈值设置是否合理，避免告警疲劳
  - 是否有告警收敛和降噪机制
  - 告警级别是否清晰（P0/P1/P2/P3）
  - 是否有告警升级机制
- **结构化日志**：日志是否是结构化的（JSON 格式），是否有 traceId/requestId 便于追踪
- **链路追踪**：是否有链路追踪系统（Jaeger、SkyWalking、Zipkin），接入是否规范
- **监控面板**：是否有清晰的监控大盘，关键指标是否一目了然
- **告警通知**：告警通知渠道是否完善，是否有值班和响应机制

---

### 4. 安全与合规（20分）

- **服务器安全基线**：
  - SSH 配置是否安全（禁用 root 登录、密钥登录、修改默认端口）
  - 防火墙配置是否合理，最小化开放端口
  - 是否有定期的安全补丁更新机制
- **密钥管理安全性**：
  - 是否使用 Vault/KMS 等专业密钥管理系统
  - 密钥是否有轮换机制
  - 禁止密钥硬编码在代码或配置文件中
- **依赖安全扫描**：是否有依赖安全扫描机制，及时发现有漏洞的依赖版本
- **备份策略完善度**：
  - 备份频率是否合理
  - 备份保留周期是否合适
  - 是否有定期的恢复演练验证备份可用性
  - 备份是否加密存储
- **网络安全**：
  - VPC/安全组配置是否合理
  - SSL/TLS 证书管理是否规范
  - 是否有 DDoS 防护等安全措施
- **合规性**：是否符合等保、GDPR 等合规要求

---

### 5. 自动化与基础设施即代码（20分）

- **基础设施即代码 (IaC)**：
  - 是否使用 Terraform/Pulumi/Ansible 等工具管理基础设施
  - 代码是否可复用、模块化
  - 是否有版本控制和变更记录
- **环境一键搭建能力**：新人是否能通过脚本快速拉起开发/测试环境
- **重复操作脚本化**：
  - 部署、数据迁移、扩容等重复操作是否脚本化
  - 脚本是否有参数化、错误处理、日志记录
  - 脚本是否有使用文档
- **运维文档完善度**：
  - 是否有 Runbook 操作手册
  - 是否有故障排查指南
  - 架构文档是否清晰完整
  - 变更流程是否有文档规范
- **变更管理**：
  - 变更是否有记录和审批流程
  - 是否有变更回滚预案
  - 变更是否在低峰期执行
- **效率工具**：是否主动开发或引入工具提升运维效率

## 运维专项扣分示例（供AI参考）

| 问题 | 扣分 | 理由 |
|------|------|------|
| 密码/密钥硬编码在配置文件中 | -10 | 严重安全隐患，零容忍 |
| Dockerfile 以 root 用户运行 | -5 | 容器安全风险，违反最佳实践 |
| 没有配置健康检查 | -4 | 服务异常无法自动发现和恢复 |
| 告警阈值设置不合理，告警风暴 | -5 | 告警疲劳，真正的问题被淹没 |
| 没有备份策略 | -8 | 数据丢失风险，严重问题 |
| 生产环境没有监控 | -10 | 盲人开车，出问题无法及时发现 |
| 所有操作全靠手动，没有脚本化 | -4 | 不可复现，容易出错，效率低下 |
| 没有环境隔离，开发测试生产混在一起 | -7 | 严重的环境管理问题，风险极高 |
| 日志没有结构化，无法检索分析 | -3 | 排查问题困难，可观测性差 |
| 没有回滚方案，出问题无法快速恢复 | -6 | 故障影响时间长，风险高 |

## 运维专项加分项

- ✅ 实现了基础设施即代码，环境管理规范化
- ✅ 主动进行安全加固，发现并修复了安全漏洞
- ✅ 开发了实用的运维工具或脚本，提升团队效率
- ✅ 建立了完善的备份和灾难恢复机制
- ✅ 文档写得清晰完善，新人可以快速上手
- ✅ 推动了 DevOps 文化建设，提升团队协作效率
- ✅ 有故障快速定位和恢复的能力，MTTR 低
